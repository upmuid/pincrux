[요건정의] 삼성전자 - 쇼핑플러스 CPS 광고 연동 개발 요건

1. 개요 및 기술 구조
* 목적: 유저가 리워드 플랫폼(쇼핑플러스)을 통해 삼성닷컴 등 삼성전자 직영몰에서 상품을 구매할 때 발생하는 매출 데이터의 실시간 연동 및 정산 검증
* 연동 방식: 
  - Inbound: 플랫폼 클릭 식별자(clickKey/pinkey) 파라미터 수신 및 쿠키/세션 저장
  - Outbound (S2S): 광고주(삼성전자) 백엔드 서버에서 플랫폼 API 서버로 결제 발생 및 변동 상태 실시간 전송 (Server to Server 방식, TLS 1.2 이상 보안 프로토콜 필수)

2. 세부 개발 요건 목록
[REQ-01] 인바운드 클릭 식별자 수신 및 유효성 유지
* 설명: 유저가 플랫폼 지면의 상품 광고를 클릭하여 삼성전자 쇼핑몰로 진입할 때, URL에 결합된 가변 고유 식별자(clickKey 또는 pinkey)를 정상 수신해야 합니다.
* 통신 규격: GET 방식 (Query String), 삼성전자 지정 인입 URL (HTTPS 적용 필수)
* 데이터 형태: clickKey / pinkey (가변 고유 난수 데이터)
* 데이터 유지: 결제를 완료하거나 장바구니에 담아두는 전 과정 동안 해당 값이 유실되지 않도록 저장 및 유지를 보장해야 합니다. 결제 직전 마지막으로 유입된 pinkey를 기준으로 성공 신호를 발송하며 해당 pinkey의 유효기간이 구매 인정기간이 됩니다.

[REQ-02] 상태별 포스트백(Postback) 통신 개발
* 설명: 광고 성공 및 정산 기준 파악을 위해 구매, 취소, 구매확정의 3가지 핵심 액션이 발생하는 시점에 삼성전자 서버에서 핀크럭스 서버로 통신(S2S)을 즉시 발송해야 합니다.
* 통신 규격: POST 방식 (Application/JSON 형태 권장)
* 대상 URL: 지정된 플랫폼 엔드포인트 URL (예: https://api.pincrux.com/postback/postback_shoppay.pin)
* 액션 구분 정의 (gubun 파라미터 또는 전송 엔드포인트 분기 처리):
  1. 구매 완료 (01): 결제 승인이 완료되어 주문이 정상 접수된 시점
  2. 결제 취소 (02): 유저의 취소/반품/환불 처리로 인해 주문이 무효화된 시점
  3. 구매 확정 (03): 정산의 최종 기준이 되는 구매확정 처리가 완료되어 더 이상 취소가 불가능한 시점
* purchase, cancel, confirm 형태도 가능 

[REQ-03] 포스트백 발송 파라미터 매핑
* 설명: 통신 시 하단 인터페이스 데이터 규격에 부합하도록 백엔드 매핑 로직을 구현합니다.
* 필수 파라미터 구성:
  - pinkey: 최초 유저 인입 시 세션/DB에 보관해 둔 클릭 고유 코드
  - appkey: 플랫폼에서 사전에 발급한 삼성전자 전용 광고 고유 식별 키값
  - gubun: 액션 상태 코드 (01: 구매, 02: 취소, 03: 구매확정)
  - amount: 실정산 대상 금액
  - traceno: 삼성전자 측 주문번호 또는 트랜잭션 고유 키값 (CS 분쟁 대응용 필수 활용)
* 상품별 구매 확정이 진행되어 분리가 필요한 경우 파라메터 추가 가능 

[REQ-04] 인프라 및 네트워크 IP 화이트리스트 설정
* 설명: 부정 송신 및 보안 어뷰징 방지를 위해 아웃바운드 포스트백을 발송할 삼성전자 측 실서버의 공인 IP(정적 IP) 정보를 플랫폼 담당자에게 공유하여 사전 등록 (미등록 IP의 접근은 게이트웨이 단에서 차단됨)

3. 개발 완료 후 검증(테스트) 시나리오

* 1단계: 클릭 연동 테스트
  - 플랫폼이 제공하는 가상 테스트 URL 발급 후, 삼성전자 테스트 환경에서 식별값 수신 및 DB 세팅 상태 모니터링
* 2단계: 액션별 포스트백 테스트
  - 테스트 식별자가 바인딩된 상태에서 가상 상품 결제(01) -> 결제 취소(02) -> 재결제 후 구매확정(03) 단계별 포스트백이 플랫폼 서버에 정합성 있게 인입되고 정상 응답을 받는지 최종 검증 후 상용 오픈
