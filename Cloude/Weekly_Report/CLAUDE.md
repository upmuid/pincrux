# 주간 업무 보고 운영 지침서 (v3.2)

> 이 프로젝트 폴더(`Weekly_Report`)가 주간 보고 봇의 단일 관리 지점입니다.
> projectbot 스킬은 사용하지 않습니다.

---

## 1. 봇의 역할과 원칙

### 1.1 핵심 역할
- **전사 레벨** 프로젝트 진행 현황 파악 및 정리
- 특정 개인 업무 일지가 아닌 **회사 전체의 프로젝트 변경 추적**
- **이번주 완료/변경 사항** 기반으로 **다음주 예정 사항** 예측 및 정리
- 수집 데이터를 **컴팩트 HTML 파일**로 출력 (데이터는 HTML 파일 내부에 인라인 `DATA` 객체로 내장 — 별도 JSON 파일 생성하지 않음)
- 확정된 보고서는 HTML로 저장하여 `weekly-report/index.html` 뷰어에서 주차별 이력 관리

### 1.2 핵심 원칙
- 왜(Why), 누가(Who), 언제(When), 어떻게(How) 변경됐는지 추적
- 이슈가 있는 내용은 **앞에 먼저** 요약 제시
- 광고주 / 매체사 기준을 기존 작성 내용 분석 후 명확히 분리
- 지난주 내용과 이번주 내용을 비교하여 **변경사항만 업데이트** (변경 없는 항목은 그대로 유지)
- 이번주 완료된 내용은 완료로 표기하고, 다음주 예정 사항을 추가로 기술

### 1.3 실행 시점
- **매주 목요일 퇴근 직전** 실행
- 코웍(Claude Cowork) Scheduled Tasks로 자동화

## 2. 데이터 수집 기간

### 2.1 수집 기간 기준 — 금~목 (7일)

```
수집 시작: 직전 주 금요일 00:00
수집 종료: 이번 주 목요일 23:59 (실행 당일)
```

> 보고서는 금요일 오전에 발송되므로, 월~목 기준으로 수집하면 직전 주 금요일
> 내용이 누락됨. 이를 방지하기 위해 수집 기준을 **금~목 7일**로 설정.

**날짜 계산 예시 (목요일 실행 기준):**

| 실행일 | 수집 시작 | 수집 종료 |
|---|---|---|
| 2026-06-25 (목) | 2026-06-19 (금) | 2026-06-25 (목) |
| 2026-07-02 (목) | 2026-06-26 (금) | 2026-07-02 (목) |

**슬랙 쿼리 예시:**
```
일일업무일지 after:2026-06-18
(after: 에 수집 시작일 전날 날짜 입력)
```

## 4. 수집 절차 (Step-by-step)

```
Step 1. 베이스라인 확보
         → 사용자가 지난주 내용을 직접 붙여넣은 경우: 해당 내용을 베이스라인으로 사용
         → 붙여넣지 않은 경우: Gmail from:me subject:주간 업무 공유 최신 1건 full content 조회

Step 2-A. 슬랙 일일업무일지 수집
         → 5개 채널 전수 검색
         → 쿼리: 일일업무일지 after:YYYY-MM-DD (수집 시작일 전날 기준)
         → 미수집 구간 있으면 진행 완료 착수 오픈 after:YYYY-MM-DD 추가 검색

Step 2-B. 슬랙 업무 협의 채널 보조 수집
         → #웹개발팀_개발요청: 신규 개발요청 및 진행률 변경 내역
         → #11_쇼핑적립: 쇼핑플러스 관련 이슈·광고주·매체 논의
         → #02_pincrux_개발업무: 배포·이슈 공유 내용
         → #with-기획파트: 기획↔개발 요건 협의 내용
         → #02_pincrux_매체사업무: 매체사 연동·테스트 협의 내용
         → 각 채널에 after:YYYY-MM-DD 기간 필터 적용

Step 3. Jira PL 프로젝트 이슈 현황 조회
         → 전체 이슈 상태별 분류
         → 수집 기간 내 업데이트된 이슈 식별

Step 4. Gmail 수신 메일 확인 (필요 시)
         → INL, 쿠팡, 키움, 버즈빌 등 협력사 관련 주요 수신 메일

         → 섹션 8 상시 모니터링 항목 반드시 상태 업데이트

Step 6. HTML 보고서 생성 및 검수 (JSON 파일은 별도로 만들지 않음)
         → 섹션 10 구조에 따라 데이터 정리 후, weekly-report/data/YYYY-WNN.html 생성
         → 데이터는 HTML 파일 하단 `<script>const DATA = {...}; renderReport(DATA);</script>`에 인라인으로 내장

         → PL WIKI > 1.회사 & 조직 > 주간 업무 보고 폴더에 페이지 생성

         → 제목 형식: [MM월 N주] 기획파트 주간 업무 보고 (YYYY-MM-DD~DD)
```

## 6. 광고주 vs 매체 vs 파트너 판별 기준

### 6.1 쇼핑플러스 CPS 광고주
쇼핑플러스에 상품/서비스를 등록하고 구매 발생 시 CPS 수수료를 지급하는 커머스 플랫폼.

> 예: 쿠팡(INL), 네이버쇼핑, 에이블리, 마켓컬리, 나스미디어(오늘의집·SSG), W컨셉, 트립닷컴, 롯데온

### 6.2 일반 광고주 (CPA·CPM·구독 등)
쇼핑플러스가 아닌 **일반 광고 집행** 형태로 진행하는 광고주.
배너·오퍼월·구독형 광고 등이 포함되며, 쇼핑 CPS 광고주와 별도로 구분하여 기재.

> 예: 아시아경제(네이버 채널 구독), 라이나생명(보험 CPS), DB손해보험, SKT, NH페이, 미니게이트
>
> 판별 기준: 상품 구매 수수료(CPS)가 아닌 광고비(CPA·CPM·구독) 기반이면 일반 광고주로 분류

### 6.3 매체사
자사 앱/서비스에 쇼핑플러스 또는 오퍼월을 **탑재하여 자사 사용자에게 직접 노출**하는 파트너.
연동·계약·테스트·오픈 프로세스를 거쳐 서비스에 적용.

> 예: 키움증권, 우리투자증권, SBI저축은행, KB PAY, 버즈빌(스타뱅킹), 부산은행,
> 카카오페이지, 라이프플래닛, KT M모바일, 롯데멤버스

### 6.4 광고 네트워크 / 애드테크 파트너
쇼핑플러스나 오퍼월을 직접 탑재하는 방식이 아니라,
**광고 인벤토리를 공급하거나 광고를 송출해주는** 애드테크 파트너.
매체처럼 자사 서비스에 SDK/API를 직접 붙이지 않음.

> 예: NBT(N+ 스토어 브랜드 펀딩), ADNET, 애드팝콘, GMO, 몰로코, AppLovin
>
> 판별 기준: 자사 앱 내 쇼핑플러스·오퍼월 지면을 직접 운영하지 않고,
> 광고 소재·예산·트래픽을 공급하거나 미디에이션 형태로 연결하면 이 분류로 처리

## 11. HTML 아카이빙 구조

### 11.1 파일 구조

```
Weekly_Report/
├── CLAUDE.md               ← 이 파일 (운영 지침)
└── weekly-report/
    ├── index.html          ← 주차 목록 사이드바 + iframe 뷰어
    ├── assets/
    │   ├── style.css       ← 공통 스타일 (보고서 HTML에서 ../assets/style.css 로 참조)
    │   └── render.js       ← 공용 렌더러 (JSON DATA → 화면 렌더링, 매주 동일 파일 그대로 참조)
    └── data/
        ├── 2026-W25.html   ← 6월 3주차 보고서 HTML (데이터는 파일 내부 인라인 DATA)
        ├── 2026-W26.html   ← 6월 4주차 보고서 HTML
        └── ...             ← 매주 HTML 파일 하나씩만 누적 (JSON 파일 생성하지 않음)
```

> 과거 주차(2026-W25~W30)는 `.json` 파일이 함께 남아있을 수 있으나 참고용 백업일 뿐, HTML 내부의 인라인 `DATA`가 항상 최신·유일한 소스다. W31부터는 `.json` 파일을 아예 만들지 않는다.

### 11.2 주차별 HTML 파일 명세

파일명: `YYYY-WNN.html`

- **HTML 파일이 유일한 데이터 소스.** 별도 JSON 파일을 만들지 않는다.
- `assets/style.css`를 `../assets/style.css`로 참조 (상대경로)
- `assets/render.js`를 `../assets/render.js`로 참조 (공용 렌더러, 매주 동일 — 새로 작성하지 않음)
- HTML 파일 본문은 섹션별 빈 컨테이너(`#section-highlights`, `#section-shopping-plus`, `#section-reward-app`, `#section-shopping-live-cps`, `#section-ad-platform`, `#section-next-week-body`, `#jira-summary`, `#jira-table-body`)만 갖고, 실제 내용은 파일 하단의 `<script>const DATA = {...}; renderReport(DATA);</script>`로 채운다
- `DATA` 객체는 섹션 10/11.3 구조를 그대로 따르는 JSON 형태의 JS 객체로 작성한다
- Jason의 리뷰·수정은 이 HTML 파일을 직접 열어 `DATA` 객체를 편집하는 방식으로 이루어진다
- `fetch`는 사용하지 않는다 (외부 `<script src>` 로드는 `file://`에서도 동작하므로 문제 없음)
- 보고서 섹션 순서: 헤더 → 주요이슈 → 쇼핑플러스 → 리워드앱 → 쇼핑라이브CPS → 광고플랫폼 → 다음주예정 → Jira파이프라인

### 11.3 인라인 DATA 객체 명세

HTML 파일 하단 `<script>` 안의 `DATA` 객체 구조 (더 이상 별도 `.json` 파일로 저장하지 않음):

### 11.4 index.html 역할
- 사이드바: 알려진 주차 목록 버튼 (하드코딩, 매주 추가)
- 우측: `<iframe>`으로 `data/YYYY-WNN.html` 로드
- `file://` 프로토콜에서도 동작 (fetch 불사용 — 각 주차 HTML은 JSON을 인라인 `DATA` 객체로 내장하고 `assets/render.js`로 렌더링)

### 11.5 새 주차 추가 절차
1. `data/YYYY-WNN.html` 생성 (인라인 `DATA` 객체 포함 — JSON 파일 별도 생성하지 않음)
2. Jason 확인 및 수정 → "수정완료" 답변 대기
3. `index.html`의 `WEEKS` 배열에 해당 주차 추가
4. Confluence PL WIKI > 주간 업무 보고 폴더에 페이지 생성 (Step 8 참조)
