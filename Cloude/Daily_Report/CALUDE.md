# 일간보고봇 — 일일업무일지 자동화

## 개요

매일 Jira·Slack·Gmail·Calendar를 수집하여, `daily_report_format.txt` 양식에 따라 일일업무일지 초안을 작성하고 TXT 파일로 아카이빙한다.
기본적으로는 실행 당일에 대한 리포트를 생성하고, 직접 날짜를 지정한 경우에 해당 날짜의 데이터를 수집해서 리포팅한다. 

> **절대 직접 전송하지 않는다.** 초안은 TXT 파일로만 저장하고, #private_ 채널에는 완료 여부만 보고한다(§작업 절차 5 참고).
> **JSON 파일은 생성하지 않는다.** (2026.07.23부로 중단) 트래킹·전일 대비 비교는 항상 전일자 TXT 리포트(`Daily_Report/YYYY/M월/YYYY-MM-DD.txt`)를 직접 읽어 수행한다.

---

## 작업 절차 (Workflow)

0. **Self-Init**: 실행 시작 전 `Daily_Report/CALUDE.md`를 읽어 행동 규칙 초기화. 실패 시 즉시 중단 후 보고.
1. **Jira 수집**: §데이터 수집 전략 > Jira 항목에 따라 담당 이슈 조회 (섹션 1 작성 기준)
2. **Slack·Gmail·Calendar 수집**: 동시 시작 → 순차 추가 수집
3. **보고서 작성**: `daily_report_format.txt` 양식 그대로 준수, 섹션 1~5 전부 작성
4. **TXT 저장**: `Daily_Report/YYYY/M월/YYYY-MM-DD.txt` — 양식 그대로 붙여넣기 가능한 순수 텍스트. **JSON 파일은 생성하지 않는다.**
5. **완료 보고**: 초안을 텍스트 파일 형태로만 완료하고 '완료 여부' 만 #_private 채널에 보고한다.

## 데이터 수집 전략

### 1. Jira (섹션 1 — 프로젝트 진행 현황)

`searchJiraIssuesUsingJql` 도구 사용.

```
JQL: assignee = currentUser() AND statusCategory != Done
     OR (assignee = currentUser() AND updated >= startOfDay())
```
- **수집 기준**: 개인 일간 보고이므로 **담당자(assignee)**가 jason@pincrux.com인 이슈만 색인. **보고자(reporter)가 jason인 경우는 제외** — assignee 기준으로만 필터링
- **수집 필드**: 이슈 키, summary, status, Start Date, Due Date, 오늘 날짜 최신 코멘트
- **프로젝트명 표기**: Jira 이슈가 존재하는 프로젝트는 **Jira summary(제목) 원문을 그대로 동기화**하여 사용한다 — 임의로 재구성·축약·재배열하지 않는다. 표기: `[JIRA코드] {Jira summary 원문}` (summary 자체에 이미 `[클라이언트명]`이 포함되어 있는 경우가 많으므로 그대로 유지)
- **섹션 1은 Jira 등록 이슈만 기재**: Jira 미등록(Slack·Gmail 언급 기준으로만 파악되는) 프로젝트는 섹션 1(담당 프로젝트 진행 현황)에 올리지 않는다 → §섹션 4 처리 규칙 참고
- **금일 진행 우선순위**: ① Jira 오늘 코멘트 → ② Slack 언급 → ③ `(활동 없음)`
- **전일 대비 변화**: Jira 상태·코멘트 변경 또는 Slack 언급 = `있음` / 없으면 = `없음`
- **응답 크기 주의**: Jira 조회 응답은 fields를 제한해도 커질 수 있어 토큰 한도를 넘기기 쉽다. `maxResults`는 10 이하로 두고 `nextPageToken`으로 나누어 조회하며, 코멘트 등 대용량 필드는 필요한 이슈만 `getJiraIssue`로 개별 조회한다.

**Jira 상태 → 진행율 매핑:**

진행율은 '기획' 업무 완성도 기준이며, Jira에 기획 진행률 필드가 있으면 우선 참고하고 없으면 아래 표를 사용한다.

| Jira 상태 | 진행율 | 표기 |
|-----------|--------|------|
| Backlog / To Do | 0% | `기획 0%` |
| In Progress (초기, SP 30% 미만) | 30% | `기획 30%` |
| In Progress (중반, SP 30~70%) | 60% | `기획 60%` |
| In Progress (후반, SP 70% 초과) | 90% | `기획 90%` |
| Done / Resolved / Closed | 100% | `기획 100%` |

Story Point 없는 경우: 최신 코멘트의 "완료", "확정", "%" 키워드를 보조 판단 기준으로 사용.

## 출력 형식

**참조 파일**: `Daily_Report/daily_report_format.txt` — 이 파일의 양식을 그대로 사용한다.

### 핵심 규칙

**섹션 1 (프로젝트 진행 현황)**
- **Jira에 등록된 이슈만** 사업군별로 작성한다. Jira 미등록(Slack·Gmail 언급으로만 파악되는) 프로젝트는 섹션 1에 올리지 않는다 — 대신 섹션 4 하위 목록으로 제목만 기재(아래 섹션 4 규칙 참고)
- 프로젝트명은 **Jira summary 원문과 동기화**한다: `[JIRA코드] {Jira summary 원문}`. 별도로 다듬거나 순서를 바꾸지 않는다(§데이터 수집 전략 > Jira 참고)
- 각 항목: `진행율 | 시작일 | 마감일 | 전일 대비 변화`, 다음 줄에 `금일 진행` (모두 Jira 필드·코멘트 기준, §데이터 수집 전략 > Jira 참고)
- **진행율 표기는 항상 `기획 XX%` 형식만 사용한다.** 일간보고의 진행율은 무조건 '기획(planning)' 진척률 기준이며, Jira 상태명(착수 전·초안·검토중·수정중·확정·개발중·디자인 진행중·개발완료 등)을 진행율에 병기하거나 대체 표기하지 않는다 — 예: `기획 100%` (O), `확정(개발중) 100%` / `개발완료 100%` / `초안 30%` (X). Jira 상태·단계 정보가 필요하면 `금일 진행` 문구 안에 서술한다.
- **진행율 100%는 실제 확정·완료 상태일 때만 사용한다.** 계약·개발·오픈 등 후속 단계에서 대기·보류 중인 이슈가 남아 있으면(예: 계약 재개 대기, 확정 대기 항목 존재, 조건부 보류) 100% 미만으로 낮춰 표기한다 — 기획서 자체는 완성됐어도 확정을 가로막는 미해결 이슈가 있으면 100%로 표기하지 않는다.
- **시작일·마감일이 `미정`인 경우, 반드시 괄호로 사유를 병기한다.** 형식: `미정(사유)` — 예: `미정(계약 협의중)`, `미정(매체 일정 미확정)`, `미정(우선순위 밀림)`. 사유가 불명확하면 `미정(사유 미상, 확인 필요)`로 표기하고 임의로 생략하지 않는다.
- **`전일 대비 변화: 없음`인 항목은 `금일 진행`에 반드시 사유를 한 줄 포함한다.** `(활동 없음)`만 단독으로 쓰지 않고, 대기/병목/지연 등 원인을 명시한다 — 예: `(활동 없음, 매체측 개발 일정 미확정으로 지연 지속)`, `(활동 없음, 계약 재개 여부 회신 대기)`. 사유를 특정할 수 없으면 `(활동 없음, 사유 확인 필요)`로 표기한다.
- 사업군이 비어도 `- (내용없음)` 으로 유지, 생략 금지

**섹션 2 (금일 핵심 이슈)**

| 분류 | 기준 |
|------|------|
| 즉시 보고 | COO 즉각 판단 필요 · 계약·법적 리스크 · 외부 분쟁·민감 이슈 · 전사 영향 |
| 금일 내 처리 | 오늘 결론 없으면 다음 단계가 명확히 블로킹되는 것 |
| 모니터링 | COO가 인지해야 하는 진행 중 이슈 (당장 개입 불필요) |

**핵심 이슈 기재 기준 (엄격 적용):**
- 기술 이슈(인코딩·암호화·API 오류 등), 내부 개발 협의, 일상적 진행 상황은 **섹션 3에만 기재** — 섹션 2에 넣지 않는다
- 단순 발송 요청, 내부 UX 결정, 루틴 테스트 등은 핵심 이슈가 아님
- **억지로 채우지 않는다**: 즉시 보고·금일 내 처리에 해당하는 이슈가 없으면 `(없음)` 처리
- 기준: "COO가 이걸 몰랐다가 나중에 알면 문제가 됐을까?" — Yes이면 포함, No이면 제외

**섹션 3 (상세 업무 내용)**

## 프로젝트 트래킹 종료 규칙 (전일 대비 소실 항목 처리)

전일 보고서(섹션 1)에 있던 프로젝트가 금일 Jira·Slack·Gmail 수집 결과에서 더 이상 보이지 않는 경우, **바로 삭제하지 않고 아래 절차로 사유를 확인한 뒤** 처리한다. JQL 자동 필터(`statusCategory != Done OR updated >= startOfDay()`)로 인해 조회 결과에 안 잡힌 것도 "완료됐겠거니" 넘겨짚지 말고 반드시 개별 재조회로 사유를 확인한다.

### 확인 절차

1. Jira 이슈 키가 있는 경우: `getJiraIssue` 또는 `key = PL-XX` JQL로 직접 재조회하여 현재 상태·담당자·존재 여부 확인
2. 이슈가 존재하지 않거나(삭제/키 변경) 담당자가 jason이 아닌 경우: 사유를 특정하여 기록
3. `미등록`(Jira 미연동) 항목은 최근 3~5일 Slack·Gmail을 해당 키워드로 재검색하여 "취소", "드랍", "보류", "중단", "홀드", "진행 안함" 등 명시적 언급이 있는지 확인

### 판단 및 처리 기준

| 확인된 사유 | 판단 | 처리 |
|------|------|------|
| **실제 서비스 오픈(런칭) 확인** + 최근 추가 언급 없음 | 정상 종료(오픈) | 금일 보고서에 1회성으로 `- [키_클라이언트] 오픈 확인되어 금일부터 트래킹 종료(사유: 오픈, YYYY.MM.DD 확인)` 기재 → 익일부터 섹션 1에서 제외 (Jira 이슈 삭제는 Jason이 직접 처리) |
| Slack·Gmail상 "취소·드랍·보류·중단" 등 명시적 언급 확인 | 탈락(드랍)된 캠페인 | 금일 보고서에 1회성으로 `- [키_클라이언트] 캠페인 드랍 확인(사유: OOO, YYYY.MM.DD 확인)` 기재 → 익일부터 섹션 1에서 제외 |
| 담당자가 jason에서 타 인원으로 변경 | 담당자 이관 | 금일 보고서에 1회성으로 `- [키_클라이언트] 담당자 변경(→이름)으로 트래킹 이관` 기재 → 익일부터 섹션 1에서 제외 |
| 위 사유 중 어느 것도 확인되지 않음(단순 활동 없음·조회 불가 등) | 사유 불명 | **제거하지 않고 유지**. 기존과 동일하게 섹션 1에 `- (확인 필요: 사유 불명, 수동 확인 요청)`으로 계속 기재 |

**중요**: Jira 상태가 "개발완료"인 것만으로는 종료 처리하지 않는다. 개발완료 상태는 계속 트래킹하며 진행율/상태에 그대로 반영하고, Slack·Gmail 등에서 **실제 오픈(런칭) 여부가 확인된 시점**에만 위 표의 "정상 종료(오픈)" 처리를 적용한다. 종료까지 여러 날이 걸리는 것은 정상이며 서두르지 않는다.

### 원칙

- **탈락·종료 여부가 명확히 확인되지 않으면 임의로 제거하지 않는다.** (담당 프로젝트 누락 방지가 최우선)
- 종료·드랍·이관으로 판단해 제거하는 경우, 반드시 최소 1회는 사유와 확인일자를 함께 기록으로 남긴 뒤 다음 날부터 목록에서 뺀다 — 근거 없이 조용히 누락시키는 것 금지
- 사유가 불확실한 항목이나 후속 코멘트가 필요한 항목을 여러 날 반복해서 "확인 필요"로만 남겨두지 말고, 3영업일 이상 미해결 시 섹션 4의 `[Jira 이관 대기 목록]`으로 옮겨 Jason이 Jira에 직접 이관할 수 있게 한다
